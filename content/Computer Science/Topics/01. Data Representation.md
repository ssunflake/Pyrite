## 1.1 Binary and Hexadecimal

### Number Systems

| System | Base | Digits used | Used by |
|--------|------|------------|---------|
| **Denary** (decimal) | Base 10 | 0–9 | Humans |
| **Binary** | Base 2 | 0, 1 | Computers (transistors) |
| **Hexadecimal** | Base 16 | 0–9, A–F | Programmers (shorthand for binary) |

> Computers use binary because transistors have two states: **on (1)** and **off (0)**.

---

### Denary → Binary Conversion

**8-bit place values:**

| 128 | 64 | 32 | 16 | 8 | 4 | 2 | 1 |
|-----|----|----|----|----|---|---|---|

**Method:** Compare denary to each place value left to right. If the denary number ≥ place value, write 1 and subtract. Otherwise write 0.

**Example: Convert 150 to 8-bit binary**

| 128 | 64 | 32 | 16 | 8 | 4 | 2 | 1 |
|-----|----|----|----|----|---|---|---|
| 1 | 0 | 0 | 1 | 0 | 1 | 1 | 0 |

150 − 128 = 22; 22 < 64 → 0; 22 < 32 → 0; 22 − 16 = 6; 6 < 8 → 0; 6 − 4 = 2; 2 − 2 = 0; 0 → 0

**Answer: 10010110**

---

### Binary → Denary

Write the place values above each bit, add up all positions where there's a **1**.

**Example: 01100011**

| 128 | 64 | 32 | 16 | 8 | 4 | 2 | 1 |
|-----|----|----|----|----|---|---|---|
| 0 | 1 | 1 | 0 | 0 | 0 | 1 | 1 |

64 + 32 + 2 + 1 = **99**

---

### Hexadecimal

| Denary | Hex | Binary |
|--------|-----|--------|
| 0–9 | 0–9 | 0000–1001 |
| 10 | A | 1010 |
| 11 | B | 1011 |
| 12 | C | 1100 |
| 13 | D | 1101 |
| 14 | E | 1110 |
| 15 | F | 1111 |

**Binary → Hex:** Group into nibbles (4 bits) from the right. Convert each nibble to one hex digit.

**Example: 11001001**
→ Group: `1100` `1001`
→ 1100 = 12 = **C**, 1001 = 9 = **9**
→ **Answer: C9**

**Hex → Binary:** Expand each hex digit to 4-bit binary.

**Why hexadecimal?** Much shorter than binary. A programmer reading memory addresses or colour codes (e.g. `#FF5733`) finds hex far easier than binary strings. Each hex digit = exactly 4 bits.

## 1.2 Binary Manipulation and Negative Numbers

### Binary Addition

Four rules:
| | |
|--|--|
| 0 + 0 = 0 | |
| 0 + 1 = 1 | |
| 1 + 1 = 10 (0, carry 1) | |
| 1 + 1 + 1 = 11 (1, carry 1) | |

**Example: Add 00110011 + 01001100**

```
  00110011
+ 01001100
----------
  01111111
```

**Overflow:** If the result needs more bits than available (e.g. 9 bits for 8-bit numbers), **overflow** occurs — the extra bit is lost and the result is incorrect.

### Two's Complement (Negative Numbers)

> Two's complement is the standard way computers represent **negative** binary numbers.

**Method to negate a number:**
1. Write the positive binary value
2. **Flip** all bits (1 → 0, 0 → 1) = one's complement
3. **Add 1**

**Example: Represent −25 in 8-bit two's complement**
- 25 = `00011001`
- Flip: `11100110`
- Add 1: `11100111`
- **−25 = 11100111**

**The MSB (most significant bit) is the sign bit:**
- 0 = positive
- 1 = negative

**Range of 8-bit two's complement:** −128 to +127

## 1.3 How Computers Represent Text, Images and Sound

### Text — ASCII and Unicode

| Encoding | Bits per character | Characters | Use |
|----------|-------------------|-----------|-----|
| **ASCII** (7-bit) | 7 (or 8 with extended) | 128 (256 extended) | English/Latin characters |
| **Unicode** (UTF-8/16/32) | 8, 16, or 32 | Over 1 million | All world languages, emoji |

- Each character maps to a **binary value**
- e.g. 'A' = 65 = `01000001`, 'a' = 97 = `01100001`
- **Character set** = the set of characters supported by a given encoding

### Images — Bitmaps

A bitmap image is a grid of **pixels** (picture elements).

| Term | Definition |
|------|-----------|
| **Pixel** | Single coloured dot; the smallest unit of an image |
| **Resolution** | Total number of pixels (width × height) |
| **Colour depth** (bit depth) | Number of bits used to represent each pixel's colour |
| **Colour depth examples** | 1-bit = 2 colours, 8-bit = 256, 24-bit = 16.7 million colours |

**File size of an image:**

$$\text{File size (bits)} = \text{image width} \times \text{image height} \times \text{colour depth}$$

$$\text{Convert to bytes: } \div 8 \qquad \text{to KB: } \div 1024$$

**Example:** 800 × 600 image, 24-bit colour
= 800 × 600 × 24 = 11,520,000 bits = 1,440,000 bytes = **1406.25 KB ≈ 1.37 MB**

### Sound — Sampling

Analogue sound is converted to digital by **sampling**:
1. Sound wave measured (**sampled**) at regular time intervals
2. Each sample's amplitude stored as a binary value

| Term | Definition |
|------|-----------|
| **Sample rate** | Number of samples taken per second (Hz) — e.g. 44,100 Hz for CD |
| **Bit depth (sample resolution)** | Number of bits per sample — e.g. 16-bit |
| Higher sample rate | More accurate sound, larger file |
| Higher bit depth | More amplitude detail, larger file |

**File size of a sound file:**

$$\text{File size (bits)} = \text{sample rate} \times \text{bit depth} \times \text{duration (s)}$$

| Unit | Value |
|------|-------|
| 1 bit | Smallest unit (0 or 1) |
| 1 nibble | 4 bits |
| 1 byte (B) | 8 bits |

## 1.5 Data Compression

> **Compression** reduces the size of a file to save storage space and reduce transmission time.

### Lossy Compression
- **Permanently removes** some data that is less important
- Cannot be fully restored to original
- Used for: images (JPEG), audio (MP3), video (MP4)
- Pros: much smaller file size
- Cons: quality loss (may not be noticeable)

### Lossless Compression
- **No data is lost** — original file can be fully restored
- Used for: text files, program files, PNG images
- Techniques: Run-length encoding (RLE), dictionary/LZ compression

### Run-Length Encoding (RLE)
- Consecutive repeated values replaced by [count, value]
- Best for images with large areas of the same colour

**Example:** WWWWWBBBBWWW
→ `5W 4B 3W` (stores 3 values instead of 12 characters)

**Limitations:** If there are very few repeated values, RLE can actually make the file *larger*.

## Key Definitions

|------|-----------|
| **Bit** | A binary digit; 0 or 1 |
| **Byte** | 8 bits |
